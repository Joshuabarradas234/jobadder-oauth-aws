AWS · eu-west-2 (London) OAuth 2.0 authorization code flow
Multi-tenant · sandbox-first
Serverless · auto-scaling architecture
Core principle One the app API application · Many JobAdder customers · One isolated, encrypted token per customer · No per-customer infrastructure
A
Customer connects
One-time, per customer — OAuth 2.0 authorization code grant
STEP 1 Customer
Clicks “Connect” in the app
Opens the authorize URL for the single the app app with the customer's chosen scopes.
GET id.jobadder.com/connect/authorize
STEP 2 JobAdder
Customer consents
Customer logs in to their own JobAdder account and approves the app's requested access.
scope: read offline_access
STEP 3 AWS
Redirect to callback
JobAdder redirects to the AWS endpoint with an auth code; API Gateway invokes the OAuth Lambda.
GET /oauth/callback?code&state
STEP 4 AWS
Exchange code for tokens
OAuth Lambda swaps the code for access + refresh tokens and the per-account API base.
POST id.jobadder.com/connect/token
STEP 5 AWS
Store this customer's tokens
The token response carries this customer's account id — store one encrypted token row for them in DynamoDB.
DynamoDB · key accountId
Per-tenant token store — DynamoDB
One item per customer, keyed by accountId: access token, refresh token, expiry, per-account api base, scope, status. Encrypted at rest with a customer-managed KMS key. Adding a customer adds a row — never new infrastructure.
KMS-encrypted Isolated rows PITR backups
B
Ongoing operation
Per-customer candidate fetch, with automatic per-tenant token refresh
STEP 6 AWS
Fetch job queued for a customer
Each SQS message is tagged with the customer's accountId. Retry ×3 then dead-letter; one customer never affects another.
SQS → Fetcher Lambda
STEP 7 AWS
Load that customer's token
Fetcher reads only this accountId's token row from DynamoDB; KMS decrypts it in memory.
DynamoDB GetItem
STEP 8 JobAdder
Call the customer's API
Request goes to the per-account API base returned at connect time. On 401, force-refresh this tenant and retry once.
GET {api}/candidates/{id}
STEP 9 AWS
Scheduled token refresh
EventBridge sweeps tenants near expiry (~50 min) and refreshes each independently — tokens never lapse.
POST /connect/token · refresh_token
Security & compliance posture
🔑
KMS encryption
All tenant tokens and the client secret encrypted at rest with a rotating customer-managed key.
🛡️
Least privilege
Requests only read + offline_access. the app reads candidate data; it does not write back to JobAdder.
📊
Full audit trail
CloudWatch logs (90-day retention) and X-Ray traces every step. No candidate PII written to logs.
🔔
Failure alerts
Dead-letter queue and refresh-failure alarms notify the operations team by email via SNS.
Tenant isolation: one customer's token can only ever read that customer's data — enforced by per-row scoping in the Fetcher. Reference multi-tenant design · illustrative only
